Domain Registration and Web Hosting

Kieren Pitts

Registering Domain Names

  • Typing address into Web browser is not enough to check if domain is already registered
  • Domain name may be registered, but may not have a Web server
  • Need to use a WHOIS tool which looks up records in the registry database
  • For .com, .org, .net, etc... domains, use InterNIC
  • For co.uk, org.uk, net.uk, ltd.uk and plc.uk, use Nominet
  • To find registrars in other countries, see IANA or Uwhois.com

Registering Domain Names

  • Don't have to register with the main registrars
  • Should choose registrar that combines good price with good service
  • I currently use Joker or DreamHost for .com/.net/.org domains and 123-reg for .uk domains.
  • Hosting providers will often offer free domain registration

Top-Level Domains

  • Up until 2000, were seven TLDs (of which only first three are available to people worldwide):
    • .com
    • .org (may in future be reserved for non-profit making organisations)
    • .net
    • .gov (reserved for US Government)
    • .edu (reserved for US educational institutions)
    • .mil (reserved for US military)
    • .int (reserved for organisations established by international treaties)

Top-Level Domains

  • Seven new TLDs were accepted in 2000:
    • .aero (for air transport industry)
    • .biz (for businesses)
    • .coop (for cooperatives)
    • .info (for anything)
    • .museum (for museums)
    • .name (for individuals)
    • .pro (for accountants, lawyers and doctors)
  • Others have been created since:
    • .jobs (for the recruitment industry)
    • .mobi (for mobile products and services)
    • .travel (for travel and tourism businesses)
    • .tel (for universal identities)
  • Other TLDs are being discussed, e.g. .xxx

Hosting

Once domain has been registered, it must be hosted. In general are three options:

  • Host site on own server on own network
    • Must support system and support network
    • Only suitable for large organisations who can take advantage of similar economies of scale to ISPs

Hosting

  • Colocate own server with ISP
    • Computer is much lower cost than network
    • Only suitable if need own box (e.g. for security or performance)
    • Still need technical support to administrate server (e.g. install security patches)
    • Cheapest price for colocation with Dilusion is £840 per year
  • (Dilusion was picked at random, so definitely not a recommendation)

Hosting

  • Host site on an ISP's server
    • Cheapest option
    • ISP takes care of technical support (backups, upgrades, redundancy, monitoring, security, etc...) leaving you to concentrate on Web publishing
    • Have someone to blame if SLA not met!
    • Cheapest price for Web hosting with Dilusion is £125 per year
  • (Dilusion was picked at random, so definitely not a recommendation)

ISP Hosting

  • Price ISP charges for hosting varies according to features needed
  • Common features that increase cost are:
    • Increasing amounts of bandwidth
    • Increasing amounts of disk space
    • Use of databases
    • Ability to run server-side scripts
    • Shell access to the server
    • Ability to run programs at scheduled times
    • Secure access to server
    • E-commerce facilities
    • Use of SSIs
    • Streaming audio / video
    • Adult content!

Choosing An ISP

  • Choice of ISP for simple Web sites that are only static HTML depends on:
    • Cost
    • Reliability
    • Technical support
    • Customer service
  • Choice of ISP for complex sites depends on more factors, for example:
    • Operating system desired
    • Scripting language desired
    • Database desired

Choosing An ISP

  • Many lists of hosting companies, e.g. UK Web Hosting Directory
  • Often better to rely on recommendation
  • Recommend:
    • For UNIX hosting, DreamHost (which this site uses)
    • For Windows hosting, ActiveHost
    • (Both are based in US)

Accessing ISP

  • Most common method of accessing ISPs is FTP (File Transfer Protocol)
  • FTP is inherently insecure because passwords are transmitted in cleartext and can be sniffed
  • If security is a priority then should access ISP securely using, for example, SSH (Secure Shell)

Common Growth Problems

  • Small Web site with one author is no problem
  • If site is larger or more than one author then many potential problems
  • If site is large then altering the user interface might require editing hundreds of pages
  • Could do global search and replace with text editor, but better to separate user interface from content using SSIs (server-side includes) or equivalent
  • If make changes to live site then users might see mistakes made during editing
  • Therefore have test site where changes are made and then reviewed (probably by other members of the team) before being uploaded to main server

Common Growth Problems

  • If have multiple authors then is a chance that one author's changes may overwrite another's
  • Need version control where file is locked whilst it is changed
  • Also allows rollback if an error was made
  • Have audit trail
  • This site uses a content management system (CMS) called CMS Made Simple

Common Growth Problems

  • CMS may take care of:
    • Staging
    • Version control
    • Auditing
    • Workflow
    • Metadata management
    • Templating
    • User management
    • Browser-based editing

Audience Growth Problems

  • As number of users increases, the upper limit of the software / hardware used might be reached
  • As load increases site may be unusable for all users (not just some)
  • Two solutions to load problems:
    • Write more efficient software (e.g. optimise database queries or generate static HTML pages from dynamic ones overnight)
    • Get better hardware / infrastructure (e.g. multiple, load-balanced servers)
  • Whilst the former is preferable, the latter is often chosen (e.g. Amazon)
  • Many sites have systems in place to deal with traffic bursts
