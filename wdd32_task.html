Formalizing a task analysis

We are now moving from specifying "the problem"/ situation (business goals, identifying users & user goals) to articulating a potential solution. The steps in constructing a formal task analysis:

  • Analyse your material and identify common processes, trends and themes. It's essential to do this step, even if you don't have much user data.
  • Describe how an identified goal may be achieved as a set of step-by-step tasks.
  • Your task analysis (or workflow) can be expressed as a bulleted list or as a flow chart. Flow charts are often better since tasks are often conditional (i.e. there can be more than one outcome).

Example task analysis #1

Task: Find and save a picture of a Shaolin Monk on the Web

  1. Go to www.google.com
  2. Type in an appropriate search term (e.g. "Image Shaolin Monk") and hit the Google Search button
  3. View the results
  4. Select results that look promising and check link for picture
  5. If no suitable picture is present, return to viewing results using "back" button on browser
  6. Repeat steps 3-5 until a suitable image is found
  7. When image is found, right click on the image and choose "save image as..."
  8. Choose the location on your hard drive where you want to save the image to.

Example task analysis #2

Task: Log on to a members-only section of your web site

  1. Enter your username and password
  2. Select "Login"
  3. If login is correct, the members home page appears
  4. If incorrect login, user is prompted for a password reminder or to return to step 1 and try again.

This is quite a small, detailed task, but there are many web processes, such as buying a book online, that have quite large discrete sub-tasks.

Why we might do task analysis

  • Test an existing process / system
  • Test the feasibility of a proposed design or prototype
  • Timed task analysis
